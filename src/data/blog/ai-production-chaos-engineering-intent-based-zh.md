*作者：Sayali Patil | 2026-04-28*

当今没有任何一款混沌工程工具能回答这样一个问题：你的上一次实验测试的是正确的东西吗？

不是"它是否在预算内？"那是 SLO 错误预算门控处理的问题。不是"系统是否存活了下来？"那是中止条件衡量的问题。问题在于，该实验是否被设计成用于验证你对系统行为的某种具体信念，以及其结果是否改变了你的团队对故障传播路径的认知。

如果你诚实的回答是"我们终止了一些 Pod，它们恢复了"，那么你进行了一次安全实验。你是否从中学到了任何有用的东西，则是当前工具没有追问的另一个问题。

本文提出了一个具体的论点：混沌工程拥有成熟的安全层和几乎为零的意图层。安全性告诉你该破坏多少。意图告诉你破坏它会教给你什么。这是两个不同的设计问题，需要不同的工具，而将它们混为一谈，是为什么混沌项目在规模化时往往积累了越来越多的脚本，却没有积累洞见的原因。

这一论点建立在我开发并申请了专利的架构（[US12242370B2](https://patents.google.com/patent/US12242370B2/en)，面向分布式系统的意图驱动混沌工程）之上，以及来自 Intuit、GPTZero、Insurance Panda、Fruzo 和 Coders.dev 的实践者们的观察——他们各自独立地诊断出了同样的结构性缺口。我将向你展示该架构，逐步介绍带有代码的数据模型，并解释为什么这是一个 AI 问题，而不仅仅是一个编排问题。

## 1. 安全层已经很好了。但它也不完整。

首先肯定当前模型的价值。SLO 错误预算框架（由谷歌 SRE 实践推广）为混沌工程提供了第一个有原则的安全机制。将实验执行与剩余错误预算挂钩，意味着你不会将故障注入到一个已经在消耗其可靠性余量的系统中 [3]。AWS 故障注入服务的停止条件、Gremlin 的可靠性分数以及 Harness ChaosGuard 的 Rego 策略，都代表了这一理念成熟的、生产就绪的实现。

这些工具回答了一个表述清晰的问题：给定系统的当前状态，现在运行实验是否安全？这个答案是可计算、可自动化且合理准确的。它们没有回答的问题同样重要：给定系统的当前状态，现在运行哪个实验最具信息量？

安全性和信息量是正交的。一个实验可以满足所有安全约束、保持在预算内、不触发任何中止、不造成任何可测量的降级，同时仍然产生任何有用的信息。如果它测试的是一个不在任何面向用户行为关键路径上的组件，你花费预算却什么都没学到。如果它重复了一种你的系统已经经历过十几次的故障模式，而没有更新你对传播路径的理解，结果也是一样。

> **核心区别：** 当实验保持在可接受的成本范围内时，它是安全的。当实验的结果更新了你对系统故障行为的模型时，它是有信息量的。这需要不同的设计标准，而只有前者拥有成熟的工具。

还有第二个结构性问题。脚本在编写时是静态的。它们编码了关于服务拓扑、流量模式和依赖行为的假设，这些假设在编写时可能是准确的，但六个月后可能会悄悄地出错。随着微服务架构每周变化，脚本与现实的偏差会不断积累。脚本仍然在运行。它在测试一个已经不存在的世界。

## 2. 实践者如何描述这一上限

以下观察来自通过 Qwoted（一个将领域专家与研究人员和记者联系起来的平台）联系到的实践者。来自不同行业的、在生产环境构建过混沌工程项目的工程师们，从不同角度汇聚到了同一个结构性缺口。

Coders.dev 创始人兼总监 Abhishek Pareek 构建分布式系统工具。他的表述对问题的诊断最为精准：

> *"我们所缺少的是对基于意图的弹性的理解。现有工具主要基于脚本，我们需要创建能够在执行实验之前对特定故障对大量微服务影响进行建模的工具。我们需要除了故障机制之外，还能理解故障背后推理的 AI。"* — **Abhishek Pareek，Coders.dev 创始人兼总监 [6]**

"推理"这个词在这里承担着真正的分量。脚本捕获的是机制：终止这些 Pod，注入这个延迟。它不捕获推理：我们进行这个实验是因为我们认为结账电路断路器应该在面向用户的错误率超过 0.1% 之前跳闸，我们想知道它是否真的会这样做。这个推理——假设——是使实验具有信息量的东西。当它只存在于工程师脑子里时，随着团队和系统的变化，它就消失了。

GPTZero CEO Edward Tian 大规模运行 AI 推理基础设施，并为所缺失的东西开发了精确的语言：

> *"当前的混沌工具在任意故障点注入故障，但不能为用户提供任何有意义的方向，告诉他们正在尝试验证什么。混沌的下一次演进将涉及针对关于弹性的特定问题，'我们的系统能否承受数据检索的降级？'或'我们是否能够容忍模型因超时而不可用？'，而不是使用一刀切的脚本。"* — **Edward Tian，GPTZero 创始人兼 CEO [7]**

"我们的系统能否承受数据检索的降级？"是一个行为假设。它命名了一个目标行为、一个故障条件以及一个隐含的成功标准。这比任何当前混沌工具所接受的输入都包含更多信息。这是设计出能回答这个问题的测试所需的最低信息量。

## 3. 意图驱动的架构

US 专利 12242370B2 描述了一个系统，其中混沌实验参数从行为意图规范推导出来，而非由工程师硬编码。以下是该架构的工作原理。

### 3.1 系统概述

该系统有四层。每一层都做了基于脚本的模型无法做的事情。实验生成器将"选择一个脚本"替换为"从你想学习的内容推导出正确的实验"。安全评估器在爆炸半径计算中添加了行为上下文。结果记录器将实验结果转化为模型更新，而不是事后分析备注。

### 3.2 意图规范

该规范是系统在生成任何实验之前所需的输入。以下是一个结账弹性测试的具体示例：

**清单 1——结账弹性实验的意图规范**

```yaml
# intent_spec.yaml
intent:
  id: exp-checkout-inv-2025-01
  target_behavior: checkout_completion
  hypothesis: >
    当库存服务经历较高的读取延迟（p99 > 500ms）时，
    结账流程在 SLO 范围内完成。
    inventory_read 上的电路断路器在
    面向用户的错误率超过 0.1% 之前跳闸。
  acceptance_criteria:
    checkout_p99_latency_ms: 400
    checkout_error_rate_pct: 0.1
    slo_budget_fraction: 0.001   # 最多消耗 0.1% 的每日错误预算
  exclusion_zones:
    - payment_auth
    - fraud_detection
    - session_management
  min_steady_state_window: 15m   # 注入前需要稳定的基线
  max_experiment_duration: 20m
```

注意这编码了传统混沌脚本所没有的内容：假设是关于系统行为的可证伪陈述，而不是对将被破坏内容的描述。验收标准以行为术语定义了"通过"的含义。排除区域和稳态窗口强制执行了大多数团队手动且不一致地处理的约束。

### 3.3 从规范到实验候选

实验生成器遍历服务依赖图，找到目标行为关键路径上的所有组件。以下是该遍历的简化 Python 示意代码：

**清单 2——使用加权依赖图的简化关键路径遍历**

```python
from typing import List, Dict
import networkx as nx

def get_critical_path_components(
    graph: nx.DiGraph,
    target_behavior: str,
    exclusion_zones: List[str]
) -> List[Dict]:
    candidates = []
    for node in nx.descendants(graph, target_behavior):
        if node in exclusion_zones:
            continue
        edge_data = graph.edges[target_behavior, node]
        candidates.append({
            'component': node,
            'call_frequency': edge_data.get('call_freq', 0),
            'degradation_sensitivity': edge_data.get('sensitivity', 0),
            'in_blast_radius_of': list(nx.ancestors(graph, node))
        })
    return sorted(
        candidates,
        key=lambda x: x['degradation_sensitivity'] * x['call_frequency'],
        reverse=True
    )
```

边权重、call_frequency 和 degradation_sensitivity 是从过去的实验和可观测性遥测（追踪、服务网格指标）中学习得到的。一个位于每次结账请求路径上且其降级历史上会传播到面向用户错误的组件排名最高。位于后台任务路径上的组件排名接近于零。

## 4. 实时安全评估：超越静态阈值

Intuit 高级工程负责人 Ishu Anand Jaiswal 指出了使安全评估真正智能（而非仅仅自动化）的组件：

> *"真正智能混沌所缺少的是一个理解实时拓扑和'弹性预算'的 AI 规划器。它应该持续估算系统还能吸收多少额外的延迟、损失或资源消耗，然后选择和排序能在该预算内最大化学习的实验，从每次运行和真实事故中更新其模型。"* — **Ishu Anand Jaiswal，Intuit 高级工程负责人 [8]**

"弹性预算"概念与 SLO 错误预算不同。错误预算衡量的是本期你已经消耗了多少可靠性。弹性预算是前瞻性的：给定系统的当前状态，它还能在实验范围之外的行为开始降级之前，吸收多少特定类型的额外压力？

表 1 展示了静态阈值门控与实时弹性评分在五个关键信号上的对比：

| **信号** | **静态阈值门控** | **实时弹性评分** |
|---|---|---|
| SLO 错误预算 | 在实验开始时检查一次 | 持续监控；如果消耗率飙升则触发中止 |
| 依赖健康状态 | 不检查 | 注入前后从服务网格读取 p99、错误率、断路器状态 |
| 爆炸半径 | 固定比例的副本（例如 10%） | 从依赖图 + 历史敏感度权重动态估算 |
| 中止信号 | 基础设施指标超过阈值 | 目标行为降级（例如结账完成率下降 > 2%） |
| 拓扑感知 | 无，脚本针对固定组件 | 实时依赖图；如果目标组件已经降级则实验重新路由 |
| 学习 | 无，运行后脚本不变 | 预测与实际爆炸半径差异更新边权重，用于未来运行 |

*表 1：静态阈值门控与实时弹性评分对比*

中止信号行是行为框架产生最具体差异的地方。意图感知实验不是在服务延迟超过阈值时停止，而是在目标行为——结账完成——降级超过验收标准时停止。不相关组件上的延迟峰值不会停止实验。结账关键路径上的延迟峰值会立即停止它，无论基础设施仪表板显示什么。

## 5. 基础设施指标无法解决的用户上下文问题

Fruzo 首席产品官 Isabella Rossi 在行为信号而非基础设施指标之上构建了混沌机制。她的观察切中了爆炸半径控制无法解决的问题：

> *"混沌工程工具通常将系统弹性视为静态属性。它们基于时间或负载阈值注入压力，这忽视了系统在一种用户上下文中可能非常脆弱，而在另一种上下文中则完全稳定的情况。注册时的数据库超时是灾难性的。同样的超时在一个可选功能中几乎不引人注意。当前工具无法做出这种区分。"* — **Isabella Rossi，Fruzo 首席产品官 [9]**

这在技术上是精确的，而不仅仅是直觉上的。注册流程中写入用户注册表的超时会终止一个会话。偏好页面中写入功能标志读取缓存的超时会静默回退到默认值。这两个事件在基础设施仪表板上看起来是相同的——数据库连接池上的超时率升高。但它们对用户的影响在数量级上有所不同。

表 2 说明了相同的故障，在相同的组件上，如何根据哪种用户行为处于活跃状态而产生截然不同的爆炸半径严重程度：

| **故障** | **组件** | **用户上下文** | **爆炸半径严重程度** |
|---|---|---|---|
| DB 写超时 | user_profile_db | 注册流程 | 严重，会话终止，用户流失 |
| DB 写超时 | user_profile_db | 偏好更新 | 低，静默回退到默认值，用户不可见 |
| Pod 终止 | inventory_service | 活跃结账 | 高，结账可能失败或超出 SLO |
| Pod 终止 | inventory_service | 夜间批量同步 | 可忽略，批量自动重试 |
| 延迟 +200ms | recommendation_api | 主页加载 | 低，异步；页面无推荐也能渲染 |
| 延迟 +200ms | recommendation_api | 结账追加销售步骤 | 中等，同步调用；在结账时增加 +200ms |

*表 2：爆炸半径严重程度取决于活跃的用户行为，而不仅仅是组件健康状况*

基于脚本的混沌工具无法填充"用户上下文"列。它不知道实验运行时哪些用户行为是活跃的。意图驱动系统可以做到这一点，因为意图规范命名了目标行为，实验生成器只考虑当前流量下该行为关键路径上的组件。

## 6. 业务信号扩展：用美元衡量爆炸半径

一旦你将实验锚定于行为而非组件，这一原则的逻辑延伸就会超出大多数 SRE 实践今天所能达到的范围。

Insurance Panda 董事总经理 James Shaffer 将整个混沌项目重建在收入信号之上：

> *"静态脚本是垃圾。它们不尊重网络的当前状态。我们将故障注入引擎直接绑定到实时业务指标，而不仅仅是服务器负载。如果活跃的报价完成量下降了哪怕两个百分点，测试就会立即停止。这是一个基于收入而非延迟的自动化熔断机制。真正智能的混沌测试缺少的不是更好的 AI 来破坏东西。而是能理解爆炸半径有多少美元价值的 AI。一个微服务失败可能在 SRE 眼里像是灾难性停机。但如果它不阻止用户购买汽车保险，谁在乎呢？智能混沌需要学会区分技术噪音和真实的财务损失。"* — **James Shaffer，Insurance Panda 董事总经理 [10]**

Shaffer 的熔断机制——由报价完成量下降 2% 触发——是行为验收标准的直接生产实现。中止信号是业务交易率，而非 p99 延迟阈值。以下是结果数据模型中的样子：

```yaml
# outcome_record.yaml
outcome:
  experiment_id: exp-checkout-inv-2025-01
  hypothesis_result: SUPPORTED   # 电路断路器如预测的那样跳闸
  abort_reason: null             # 实验完整运行
  # 行为信号（验收标准）
  checkout_p99_latency_ms: 312   # 通过：< 400ms
  checkout_error_rate_pct: 0.04  # 通过：< 0.1%
  checkout_completion_rate_delta: -0.3%  # 通过：< 2% 阈值
  # 爆炸半径：预测与实际
  predicted_blast_radius:
    - inventory_read_service
  actual_blast_radius:
    - inventory_read_service
    - cart_service   # 已发现的依赖，不在图模型中
  budget_consumed_pct: 0.00083
  # 模型更新信号
  graph_updates:
    - add_edge: [checkout, cart_service]
      sensitivity_weight: 0.34
  blast_radius_prediction_error: 0.34
```

这个记录中最有价值的一行是已发现的依赖：cart_service 不在图模型中，但实验揭示了它响应 inventory_read 降级。这个更新向前传播——下一个结账实验将在其爆炸半径评估中包含 cart_service。这就是系统对自身的模型如何随时间改善，无需人工策划。

## 7. 为什么这是 AI 问题，而不仅仅是编排问题

此时合理的反对意见是，上面描述的一切听起来像是工程工作——依赖图遍历、阈值比较、结构化日志记录。我们真的需要 AI 来做这些吗，还是只需要更好的"管道"？

管道处理的是确定性决策：如果消耗率超过 X，则中止。如果延迟超过 Y，则停止。这些是当前工具实现的护栏。它们是有价值的，在已知假设下是封闭的。需要学习模型的问题是那些决策空间无法枚举的问题：

- **新拓扑上的爆炸半径预测。** 预测故障对未直接目标的组件的二阶影响，需要从过去实验的行为模式中进行泛化。你无法在编写时枚举所有可能的服务图。

- **假设生成。** 将"在库存降级下测试结账弹性"转化为按预期信息量排序的故障类型列表，不是规则执行。它需要对服务行为之间语义关系的推理。

- **敏感度权重学习。** 依赖图中的边权重不是静态属性。它们随着流量模式、缓存行为和部署变化而变化。它们需要从实验结果中持续学习。

- **实验中的异常归因。** 当实验中多个信号同时移动时，确定哪个移动是由注入的故障引起的，而哪个是预先存在的条件引起的，这需要一个反事实模型。这是一个因果推理问题。

最后一点是该领域距离解决方案最远的地方。自适应混沌工具在关联信号方面表现尚可，但无法解释为什么特定故障以给定的方式在特定拓扑中级联 [4]。建立这种能力需要当前任何混沌工具都没有尝试过的东西：一个可以从实验结果更新并用反事实查询进行询问的故障传播因果模型。

## 8. 认真对待反驳意见

成熟的团队已经在撰写假设陈述了。Basiri 等人（2016）的混沌工程原则要求在注入之前定义稳态行为 [2]。Netflix、谷歌和 Intuit 运行有纪律的项目，工程师在运行实验前记录他们预期会发生什么。那么"意图驱动的混沌工程"只是在描述谨慎从业者已经在做的事情吗？

这一反对意见部分正确。成熟的团队确实保存假设陈述。问题在于，他们把这些陈述保存在文档中，而不是工具中。假设存在于 Notion 页面中。执行实验的混沌工具无法访问它。这造成了四个具体的缺口：

- 工具无法验证实验设计是否真正测试了所陈述的假设——文档意图与配置故障之间的不匹配永远不会被发现
- 工具无法根据相对于假设的实时系统状态调整实验——无论当前条件是否使测试有意义，它都会运行
- 工具无法基于预测与实际爆炸半径之间的差异更新依赖模型——那个信号消失在事后分析文档中
- 工具无法防止同一假设被重复测试——脚本库在增长，洞见却没有

"团队手动做这件事"和"工具使这件事可计算"之间的区别，就是一种随团队扩展的实践和一种不能扩展的实践之间的区别。当写了假设陈述的工程师离职时，意图也随之消失了。当系统拓扑发生变化时，假设可能不再对应于任何真实的实验设计，而没有任何机制能捕获这一点。

## 9. 该领域需要构建的三件事

架构已经存在。它所依赖的安全原语已经成熟。它所需要的可观测性基础设施已经广泛部署。该领域与其所需状态之间还有三个具体缺口。

**缺口 1：标准意图规范模式**

每个做假设驱动混沌工程的团队都使用自己的格式——Notion 模板、运行手册章节、JIRA 工单类型。这些都不是混沌工具机器可读的。清单 1 中的五个字段（target_behavior、hypothesis、acceptance_criteria、budget_fraction、exclusion_zones）捕获了基本结构。标准化这个模式——类似于 OpenAPI 标准化服务接口描述的方式——将让工具能够摄取、验证和根据假设行动，而不是忽略它们。

**缺口 2：结构化实验结果数据**

爆炸半径预测需要训练数据。目前几乎没有团队以结构化、可查询的格式记录实验结果。结果存在于 Slack 线程和事后分析文档中。清单 4 中的结果模式是一个起点。对现有混沌工具进行自动发出结构化结果的改造——并将其与依赖图一起存储在可查询的格式中——将产生预测模型所需的训练信号。

**缺口 3：假设质量评估**

混沌项目目前根据覆盖率（测试了多少服务）和存活率（系统是否坚持住了）进行评估。这两者都不衡量实验是否具有信息量。一个假设质量分数——这次运行的结果改变了团队对系统的认知吗，改变了多少？——将给从业者提供一个改善实验设计的信号，而不仅仅是积累脚本。这些都不需要新的研究。它们需要该领域就表示方式达成一致，并投资于使从实验中学习可计算而非轶事性的数据基础设施。

## 结论

[混沌工程](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=6519940)拥有正确的安全原语。它缺少的是一种同样有原则的信息量方法。没有意图层，混沌项目往往走向两种失败模式：反复测试相同事物的脚本，以及在预算内运行却没有产生任何值得学习内容的实验。

本文所讨论的意图驱动架构不会取代该领域已经构建的安全机制。它增加了一层使这些机制更有意义的东西——将实验植根于操作员实际想要学习的内容，从行为规范而非工程民间智慧推导实验，并积累一个随每次运行改善的系统故障动态模型。

这个缺口是真实的、结构性的、可以解决的。问题在于该领域是否构建基础设施来弥合它，还是继续编写脚本。
